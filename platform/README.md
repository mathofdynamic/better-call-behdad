# Platform bindings

Behdad's **portable core** lives in `agents/`, `scripts/`, `config/`, `schemas/`, `SKILL.md`, and
`AGENTS.md`. Those files know nothing about any specific agent host. This `platform/` directory
holds the thin, host-specific glue that lets the same audit run on different tools. **Never put
audit logic here** — only orchestration wiring. When behaviour needs to change, edit `agents/` and
re-run the generators.

## What's portable vs host-specific

| Concern | Portable core | Claude Code | Codex |
|---|---|---|---|
| Agent instructions | `agents/**` (source of truth) | thin `.md` pointers | thin `.toml` pointers |
| Deterministic layer | `scripts/*.py` (stdlib) | same | same |
| Rules & schemas | `config/**`, `schemas/**` | same | same |
| Model / effort tiering | — | frontmatter `model:` | `model_reasoning_effort` |
| Structured return | prompt-specified | prompt-specified | `output_schema_ref` (enforced) |
| Fix gate | — | `PreToolUse` hook, default-deny (exit 2) | none — read-only sandbox + Codex approval prompts only |
| Invocation | "audit with behdad" | `/behdad` command | agent invocation |

## Claude Code (`platform/claude/`)
- `agents/behdad-*.md` — subagent definitions (inspectors = Sonnet, critic = Opus). Generated by
  `agents/_generate_bindings.py`. Install by copying into the project's `.claude/agents/` (or point
  Claude Code at this dir).
- `hooks/gate.py` + `hooks/settings.snippet.json` — the execution-layer fix gate. Merge the snippet
  into `.claude/settings.json`. The hook is inert unless `BEHDAD_ACTIVE=1`, so it's safe to leave on.
- `commands/behdad.md` — the `/behdad` slash command. Copy into `.claude/commands/`.
- The **manager** is the main agent (driven by `SKILL.md` + `agents/manager.md`), so it spawns
  inspectors at depth 1 — no nested-subagent depth issues.

## Codex (`platform/codex/`)
- `agents/behdad-*.toml` — TOML agent bindings (inspectors = medium effort, critic = high),
  `sandbox_mode = "read-only"` during inspection, `output_schema_ref` pointing at the finding
  schema so returns are validated. Generated by `agents/_generate_bindings.py`.
- Codex reads `AGENTS.md` from the repo root for project context (already portable).
- The Codex agent-definition schema is still evolving — verify field names against your installed
  Codex version; the generator is the one place to adjust them.

## Regenerating
```bash
python platform/claude/agents/_generate_bindings.py
python platform/codex/agents/_generate_bindings.py
```
Both read the roster + tiers from their generator script and write pointer files that defer to
`agents/`. Change tiers/roster there, not in the generated files.
